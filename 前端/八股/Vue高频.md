# 什么是MVVM模式，在vue中如何体现
MVVM 模式有三个核心部分，分别是模型层 Model, 视图层 View, 和视图模型层ViewModel。模型层负责数据和业务逻辑，不关心数据如何被展示。视图层负责界面展示和交互，不包含业务逻辑。视图模型层连接了模型层和视图层，具有双向绑定的能力，因此当`Model`中的数据改变时会触发`View`层的更新，`View`中由于用户交互操作而改变的数据也会在`Model`中同步。他的优势是降低耦合，让 view 和 model分离，可以独立开发。viewmodel 的逻辑可以被不同的view复用，提高开发效率
在vue中
model 层是 vue 中的数据对象，data, reactive, ref 这些响应式数据对象
view 层是模板 template 生成的 dom
viewmodel 层在 vue3 里是`setup` 函数中返回的一系列响应式绑定和逻辑组合构成的逻辑层。
如何实现依赖追踪，视图自动更新：
vue是使用 proxy 实现依赖追踪的。 使用 ref 或 reactive 定义数据时，vue会使用proxy对原对象进行包装。vue 内部有一个全局变量 activeEffect 指向当前执行的渲染函数或计算属性，当组件渲染时会触发属性的 get 拦截，vue 就将当前的activeEffect存入一个 `targetMap` 里，这是一个WeakMap,它记录了依赖关系。之后当数据变化时， Vue 会去 `targetMap` 中根据当前被修改的“对象”和“键”，找到之前存进去的所有副作用函数，依次执行这些函数，从而触发组件重新渲染或执行 `watch/computed`
进阶: 为什么targetMap要使用 WeakMap 而不是普通 Map
普通 Map：对键持有的是强引用。只要 Map 存在，且键还在 Map 中，即使外部没有任何变量指向这个键对象，垃圾回收机制（GC）也不会回收它。WeakMap：对键持有的是弱引用。如果一个对象只被 WeakMap 引用，垃圾回收机制会回收它。因此如果用Map就可能内存泄漏，就比如组件已经卸载，但是GC也不回收被Map引用的数据对象，应用运行越久就会越卡
# vue2 和 vue3的区别
响应式实现：
Vue 2 (Object.defineProperty):通过递归遍历数据的属性,重写属性的 `getter` 和 `setter` 来拦截读写操作。无法监听对象属性的新增和删除，也无法监听数组索引和长度的变化。Vue 3 (Proxy)基于 ES6 的 `Proxy` 。直接监听整个对象而非属性，针对该对象的操作都会先经过 Proxy 的拦截器，因此支持支持数组和对象的新增/删除；同时由于是“懒处理”（用到时才代理），初次渲染性能大幅提升。

API风格: 
Vue 2 (Options API)：采用“选项式”组织代码，将逻辑强制拆分为 `data`、`methods`、`computed` 等选项。在简单组件中这很清晰，但在复杂组件中，同一个业务逻辑会被打散在不同选项中，维护困难。Vue 3 (Composition API)：引入了“组合式 API”（配合 `<script setup>` 语法糖）。我们可以按功能逻辑来组织代码，将相关的状态、方法、计算属性聚合在一起（即 Composables）。让逻辑变得更灵活

性能：
Vue 3 的所有 API（如 `ref`, `reactive`）都支持 Tree-shaking。如果未使用，打包时会被自动移除，使得核心包体积更小。编译时优化：Vue 3 引入了静态提升 (Hoisting)、补丁标志 (PatchFlag)和事件监听缓存。编译器在构建阶段就能分析出哪些是静态节点、哪些是动态节点。在 Diff 算法运行时，Vue 3 可以直接跳过静态内容的比对，只针对带有标记的动态节点进行操作，这使得渲染速度相比 Vue 2 有了显著提升。
# 生命周期钩子函数
初始化，vue2是beforeCreate 和 created，vue3一般用setup()。一般用来数据初始化，定义响应式变量。

挂载，onbeforeMount 和onMounted。
`onBeforeUpdate`：数据已更新，但 DOM 还没开始重新渲染。适合在更新前访问现有的 DOM 状态（比如记录滚动条位置）。
`onUpdated`：
 DOM 已经根据最新数据完成了更新。一般处理更新后的 DOM 相关逻辑。但是避免在此钩子内直接修改响应式数据，否则可能导致无限循环更新。

卸载 onBeforeUnmount ，onUnmounted
`onBeforeUnmount`：
组件即将被销毁，但实例和 DOM 依然可用。用于清理工作，如清除定时器（`setInterval`）、销毁插件实例、解绑手动监听的事件，防止内存泄漏。
`onUnmounted`
组件已彻底销毁。最后的清理逻辑。

进阶：为什么异步请求通常放在 `onMounted` 阶段。
很多异步请求的后续逻辑是依赖真实 DOM 的。如果在 `setup` 阶段执行，此时组件仅仅是完成了响应式数据的初始化，真实的 DOM 节点还没有生成。直接操作 DOM 会导致 `undefined` 错误或插件初始化失败。并且将请求放在 `onMounted`，意味着 Vue 可以先完成组件的首屏骨架渲染。用户能先看到基础 UI（如 Loading 状态或侧边栏），随后数据异步返回后再填充内容。
# 响应式原理怎么实现
Vue3 的响应式原理就是通过 Proxy 对对象或数组进行代理拦截，通过 track 收集依赖，trigger 触发更新，ref 用于基本类型或单值，reactive 用于对象或数组，实现数据变化自动驱动视图更新，支持深层属性和组合式 API。
核心三个步骤：
代理对象 (Proxy)
    使用 `reactive()` 函数创建响应式对象时，Vue 3 会返回一个由 `Proxy` 包装过的对象。这个代理可以拦截对象的所有操作（如 `get`, `set`, `deleteProperty` 等）。
    
依赖收集 (在 `get` 拦截器中调用 `track`)
    当访问响应式对象的属性时，会触发 `Proxy` 的 `get` 拦截器。
     在 `get` 拦截器中，Vue 会调用 `track(target, key)` 函数。`track` 的作用是“追踪”，它会将当前正在执行的副作用函数（`effect`，相当于 Vue 2 的 `Watcher`）与这个属性（`target` 对象的 `key` 属性）建立关联。
     Vue 3 使用一个 `WeakMap`（称为 `targetMap`）来存储这种依赖关系，结构为 `targetMap -> depsMap -> dep`，实现了更精细化的管理。
     
触发更新 (在 `set` 拦截器中调用 `trigger`)
    当修改响应式对象的属性时，会触发 `Proxy` 的 `set` 拦截器。
    在 `set` 拦截器中，Vue 会调用 `trigger(target, key)` 函数。`trigger` 的作用是“触发”，它会从 `targetMap` 中找到所有依赖于该属性的 `effect` 函数，并执行它们，从而触发视图更新。
# computed 和 watch
Computed 用于计算状态，依赖不变时自动缓存，watch 用于监听数据变化执行副作用，支持深度监听和立即执行。两者结合 reactive/ref 可实现高效状态管理。
computed
* 是计算属性，它基于响应式依赖进行缓存。只有依赖项改变时才重新计算，否则返回缓存。它在底层维护了一个 `dirty`（脏检查）开关：初次读取执行计算，并将结果缓存，`dirty` 置为 `false`。再次读取如果依赖的响应式数据没变，直接从内存拿缓存值，不执行函数。依赖变了：触发 `trigger`，将 `dirty` 置为 `true`。下次读取时，发现是“脏数据”，才会重新计算。在处理大数据量运算时，保证了性能
* 场景：列表过滤或排序，根据用户的输入动态计算显示值

watch
* 是侦听器，监听 reactive/ref 数据，变化时执行回调。它没有缓存，只要监听的目标变了，回调就会执行，可以执行异步逻辑，支持深度监听(设置deep属性)，可以设置立即执行一次(设置immediate属性)
* 场景：发送接口请求，数据变化触发缓存或 localStorage
# 虚拟 DOM 与 Diff算法
Vue 使用虚拟 DOM 将真实 DOM 抽象为 JS 对象，通过 Diff 算法对比新旧节点，只更新变化部分，从而提高性能。v-for 使用 key 可以保证列表节点复用和状态稳定，同时提升渲染效率。
**虚拟DOM**
解释：虚拟 DOM 本质上是一个 普通的 JS 对象，它用简单的属性来描述真实的 DOM 结构。当组件渲染时，框架并不会立即操作真实的 DOM，而是先在内存中创建一棵由这些 JS 对象构成的“虚拟 DOM 树”。
好处： 直接操作 DOM 的代价非常高，因为会触发浏览器的重排（Reflow）和重绘（Repaint）。而操作 JS 对象极快。并且可以实现跨平台，由于虚拟 DOM 与真实环境解耦，这使得一套代码可以渲染到浏览器，手机或者服务端。
**Diff 算法**
解释： 当数据发生变化时，框架会生成一棵新的虚拟 DOM 树。Diff 算法的作用就是对比新旧两棵虚拟 DOM 树，找出它们之间的差异（即“最小变更集”）。

核心
* 同级比较
    Diff 算法只会对同一层级的节点进行比较，不会进行跨层级的对比。如果发现某个节点在不同层级间移动，框架会简单地销毁旧节点并创建新节点，而不是尝试移动它。
    核心：前序比较和后序比较，预处理掉开头和末尾的未发生变化的节点 => 求LCS => Map下标映射 => 求LIS => 前驱节点回溯 => 只移动LCS外的节点
* 类型不同则直接替换
    如果新旧节点的类型不同（例如，从 `<div>` 变为 `<span>`，或从组件 A 变为组件 B），框架会直接销毁整个旧节点及其所有子节点，然后创建并挂载全新的节点。
* 列表节点的 Key 优化
    这是最关键的一点。在渲染列表时，为每个节点提供一个唯一且稳定的 `key` 属性（如数据的 `id`），可以帮助 Diff 算法准确地识别出哪些节点是新增的、删除的或仅仅是移动了位置。如果没有 `key`，框架会默认按索引顺序进行比较，这可能导致不必要的重新渲染和状态错误。

key的作用：`key` 是节点的唯一标识（Identity）。它的核心作用是最大化复用已有节点，从而减少 DOM 操作的开销。
* 通过 `key`，Vue 可以准确地识别出哪些节点是新增的、删除的或仅仅是移动了位置。这样，在更新列表时，Vue 只需要对真正发生变化的节点进行操作，最大限度地复用已有的 DOM 元素，提升了渲染性能。
* 当一个组件或 DOM 元素（比如包含输入框等用户交互元素的）被复用时，它的内部状态（如输入框的值、组件的 `data` 数据）会被保留。使用稳定的 `key`，Vue 能确保状态与正确的数据项绑定，避免错乱。
为什么不用index作为key: `key` 的作用是给虚拟 DOM 节点提供一个稳定且唯一的身份标识，而index是不稳定的，比如列表发生增删插入的时候, index就会变，diff算法就会误判。性能就会变差，或者状态混乱


# 组件通信
组件通信根据层级和用途选择方案，父子通信用 props/emit，跨层用 provide/inject，跨模块用 Pinia，全局状态统一管理可提高可维护性。
Pinia 是 Vue 官方推荐的状态管理库
**父子通信**
父传子 props, 子传父emits
- Props (父传子)：父组件通过属性（`props`）向子组件传递数据。这种方式是单向的、响应式的，数据从父组件流向子组件，子组件如果要修改父数据需要提供回调。子组件不应直接修改 `props`。
- Emits (子传父)：子组件通过触发自定义事件（`emit`）向父组件发送消息或数据。这使得子组件可以与父组件进行交互，而无需直接访问父组件的状态。
- 场景：父组件向子组件传递配置、数据；子组件向父组件报告操作结果（如表单提交、按钮点击）。
**跨层通信**
直接使用 `provide` 和 `inject` 函数。
祖先组件使用 `provide` 函数，接收两个参数：注入的 `key`（通常是 Symbol或者InjectionKey(ts)）和要传递的 `value`。
后代组件使用 `inject` 函数，传入相同的 `key` 即可获取数据。
为了保证数据是响应式的，`provide` 传递的必须是响应式对象（如 `ref` 或 `reactive` 的返回值）。如果传递的是 ref的value，后代拿到的就是静态值，无法响应变化。如果你传递ref，后代拿到的就是响应式引用。
使用 Symbol 作为 Key：防止字符串拼写错误或命名冲突。
配合 TypeScript：使用 `InjectionKey` 接口来确保类型安全。
`export const UserKey: InjectionKey<User> = Symbol('User')`
**全局状态管理**

在 Pinia 中，我们通过 `defineStore` 来定义一个 Store，它主要包含三个核心部分，分别对应组件中的概念：
* State（状态）：相当于组件的 `data`。它是响应式的数据源，通常是一个返回对象的函数。
*  Getters（计算属性) ：相当于组件的 `computed`。用于对 State 进行计算和派生，具有缓存特性。
* Actions（操作）：相当于组件的 `methods`。 既支持同步也支持异步，并且可以直接在 Actions 中修改 State
如果我们直接从 Store 中解构属性（例如 `const { name } = userStore`），响应式会丢失。这是因为解构切断了与 Store 内部 Proxy 对象的联系。需要使用`storeToRefs` 函数进行结构(Actions不需要使用storeToRefs也可以，函数本身不会丢失上下文)
# 动态组件与异步组件
动态组件通过 `<component :is>` 可以在同一位置渲染不同组件，异步组件可按需加载，减少初始包体积，提高性能。
**动态组件**
动态组件允许你在同一个地方(挂载点)，根据逻辑动态地切换不同的组件。语法是`<component :is="activeTab" />`, 一般用来写tab 页、条件渲染不同表单
进阶：用 `<KeepAlive>` 包裹 `<component>`。这样组件切换时不会被销毁，而是进入缓存状态。比如用户从聊天界面进其他页面，再切回来时保留聊天记录
**异步组件**
defineAsyncComponent, 异步组件使得组件仅在需要时才通过网络请求加载，从而实现懒加载，优化首屏性能。
```js
import { defineAsyncComponent } from 'vue'
const AsyncComp = defineAsyncComponent(() => import('./MyComp.vue'))
```
# 插槽
默认插槽
子组件写一个`<slot>`，父组件往里填内容。如果父组件没填，子组件可以显示默认内容。一般用在简单内容比如一个`button`，里面显示什么图标或者文字
具名插槽
当一个组件有多个地方需要自定义时，就通过`name`属性给 `<slot>` 起名字。父组件通过`template` 加上`v-slot:`或`#`指定内容插入到哪个插槽中。一般用在复杂布局
作用域插槽
用来将子组件内部的数据传递给父组件。子组件在slot中绑定属性(v-bind)，父组件`#/v-slot`接收这些数据，然后利用数据进行渲染
# v-if 和 v-show
实现方式
`v-if`：是惰性渲染。只有当条件为 `true` 时，才会渲染这个元素，并将其插入到 DOM 树中。条件变为 `false` 时，会从 DOM 中移除并销毁。
`v-show`：是CSS 样式切换，元素始终会被渲染并保留在 DOM 中。它通过动态修改元素的 CSS `display` 属性来控制。(条件为 `false` 时，`display` 被设置为 `none`。)

渲染开销
v-if: 条件切换时会销毁/重建 DOM，开销大。但是变化少的场景开销小，因为初始为 false 就不会进行渲染
v-show: 初始渲染时就会渲染 DOM，切换显示是修改css进行隐藏, 开销小。所以适用于切换频繁的场景

# v-for 为什么需要 Key
`key` 的核心作用是给每个节点一个唯一的身份标识，帮助 Vue 的 Diff 算法更高效、更准确地识别节点，避免大量的DOM重排或状态错乱。
# Vue 性能优化手段

缓存：
* v-once,：初次渲染后，该节点会被视为静态内容之后永远不要追踪它的变化.`eg: <span v-once>说明文档：{{ msg }}</span>`
* v-memo: 通过一个依赖数组来手动控制组件何时重新渲染。只有当依赖数组中的某个值发生变化时，Vue 才会更新 DOM。如果数组没变，就直接复用上一次的缓存。
 `eg:<div v-memo="[dependency1, dependency2]"> </div>`
* 组件状态缓存 (`<KeepAlive>`)，对于频繁切换的组件（如 Tab 标签页、列表页与详情页之间的跳转），可以用 `<KeepAlive>` 组件将其包裹起来。`<KeepAlive>` 会将组件实例缓存起来，而不是在切换时销毁。
* 计算属性缓存 (`computed`)，`computed` 是基于其响应式依赖进行缓存的，只有当依赖的数据发生变化时，它才会重新计算；否则，直接返回缓存值

懒加载
* 路由懒加载：配置 Vue Router 时，不使用同步 `import`，而是使用动态导入语法 `() => import('...')` 来定义 `component` 属性。
* 组件异步加载：defineAsyncComponent, 异步组件使得组件仅在需要时才通过网络请求加载，从而实现懒加载，优化首屏性能。
避免深层响应式对象
* `shallowRef` / `shallowReactive`：当处理一个庞大的对象时，如果只需要监听其顶层属性的变化，可以使用 `shallowRef` 或 `shallowReactive`。它们只会将对象的最外层属性变为响应式，而内部嵌套的对象则保持“惰性”，不会被转换为响应式代理
* `Object.freeze()`:对象第一层属性不可扩展，配置，写。不需要响应的数据可以用 Object.freeze() 冻结。Vue 发现对象被冻结了，就会跳过响应式劫持
* `markRaw`: 不需要响应的对象可以用 markRaw。数据完全可修改，只是不能被代理。
