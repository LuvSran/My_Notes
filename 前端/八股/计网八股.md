# GET 和 POST 的区别
| 对比维度 | GET 请求 | POST 请求 |
| :--- | :--- | :--- |
| 数据位置 | URL 末尾（查询字符串 `?key=value`） | 请求体（Request Body）中 |
| 数据可见性 | 明文暴露在地址栏，会被浏览器历史记录保存 | 隐藏在请求体中，对用户不可见 |
| 数据长度限制 | 受限于浏览器和服务器对 URL 长度的规定 | 理论上无限制（但服务器可能有配置限制） |
| 安全性 | 较低，不适合传输敏感信息（如密码） | 更隐蔽，适合传输敏感信息(但也是明文传输，真正的安全需要HTTPS) |
| 缓存 | 可被浏览器主动缓存，可收藏为书签 | 默认不会被缓存，也不能被收藏为书签 |
| 幂等性 | 幂等：多次执行相同操作，结果一致 | 非幂等：多次提交可能会产生不同结果（如创建多个资源） |
| 主要用途 | 获取/查询数据（Read） | 提交/创建/修改数据（Create/Update） |
# TCP/IP四层
* 应用层： 直接面向用户和应用程序，为应用程序提供网络服务接口。HTTP/HTTPS，DNS
* 传输层： 这一层负责提供端到端（进程到进程）的通信服务，通过端口号来区分不同的应用程序。提供数据传输服务。TCP,UDP
* 网络层: 处理数据包在网络中的路由和转发。使用 IP 地址唯一标识网络中的主机。根据 IP 地址，,将数据包从源主机发送到目标主机。IP,ARP
* 网络接口层: 处理与物理网络的接口细节。WIFI
# TCP 和 UDP区别
|对比维度|TCP (可靠的快递员)|UDP (随性的广播员)|
|---|---|---|
|**连接性**|**面向连接** (传输前需三次握手)|**无连接** (直接发送数据)|
|**可靠性**|**高** (保证数据不丢、不错、有序)|**低** (尽最大努力交付，不保证可靠)|
|**传输速度**|**较慢** (有确认、重传等额外开销)|**极快** (无额外开销，实时性强)|
|**数据顺序**|**流式传输,保证顺序** (按发送顺序交付) |**数据包传输，不保顺序** (可能乱序到达) |
|**头部开销**|**大** (最小20字节)|**小** (固定8字节)|
# HTTP 1.1 和 2 的区别
1. 多路复用：允许在 单个TCP连接 上并行交错发送多个请求和响应 ，解决队头阻塞，提升传输效率。
2. 头部压缩：引入了 HPACK 压缩算法，减少冗余头部数据，节省带宽。客户端和服务器同时维护头信息表，所有字段都会存入这个表，生成一个索引号，以后就不发送同样字段了，只发送索引号，这样就能提高速度了
3. 二进制协议：采用 二进制帧 替代 HTTP/1.1的 文本协议，使得解析更快更高效。
4. 流优先级：允许按权重和依赖关系优先传输关键资源，优化用户体验。
5. 服务器推送：服务器 主动推送 资源给客户端，而不需要客户端明确请求，减少额外请求延迟
6. 数据流：HTTP/2 使用了数据流的概念，因为 HTTP/2 的数据包是不按顺序发送的，同一个连接里面连续的数据包，可能属于不同的请求。因此，必须要对数据包做标记，指出它属于哪个请求。HTTP/2 将每个请求或回应的所有数据包，称为一个数据流。每个数据流都有一个独一无二的编号。数据包发送时，都必须标记数据流 ID ，用来区分它属于哪个数据流。

# HTTP2解决了哪些问题，仍然有什么问题没解决,HTTP3如何解决的
### 解决的问题
1. 应用层队头阻塞
    - **问题**：HTTP/1.1 即使使用长连接，也必须串行处理请求。一个慢请求会阻塞后续所有请求。
    - **解决**：通过 **多路复用 **，允许在**单个 TCP 连接**上并发、交错地传输多个请求和响应的“帧”，彻底消除了应用层的队头阻塞。
2. 头部冗余 
    - **问题**：HTTP/1.1 每次请求都要携带大量重复的头部信息（如 Cookie, User-Agent），浪费带宽。
    - **解决**：引入 **HPACK 头部压缩算法**，利用静态表和动态表，用索引号代替重复的头部字符串，大幅减小了头部体积。
3. 连接开销大
    - **问题**：为了并行下载资源，浏览器需要为同一个域名打开多个 TCP 连接（通常6-8个），每个连接都需要三次握手，消耗服务器资源。
    - **解决**：多路复用使得**一个 TCP 连接**足以高效处理所有通信，显著减少了连接数和建立连接的开销。
4. 低效的文本协议 
    - **问题**：HTTP/1.1 是基于文本的，解析效率低且容易出错。
    - **解决**：采用 二进制分帧 ，将消息拆分为更小、更易解析的二进制帧，提高了处理效率和可靠性。
### 仍然解决不了的问题
* **TCP 队头阻塞：** 在一个承载了多个 HTTP/2 流的 TCP 连接中，如果任何一个数据包在网络中丢失，TCP 会暂停，直到丢失的包被重传成功。
* **握手延迟高：** 建立一个安全的 HTTPS 连接需要先完成 **TCP 三次握手 (1-RTT)**，再完成 **TLS 握手 (通常 1-2 RTT)**，总共需要 **2-3 个 RTT (Round-Trip Time)** 才能开始传输数据。
* TCP连接由 IP 和 端口 唯一标识，当网络环境变化(wifi 换成 流量) ，原有的TCP连接就会中断，需要重新建立连接
### HTTTP3解决的问题
- **UDP**没有TCP的队头阻塞问题，因此通过使用QUIC协议实现的**HTTP/3**能避免TCP连接中的队头阻塞。
- **多路复用**：QUIC 在**传输层**就实现了**多路复用**，并且每个流是**完全独立**的。解决了TCP连接中的队头阻塞问题。
- **更低的连接延迟**：QUIC通过0-RTT连接重用，减少了连接建立的延迟。
- **更好的移动网络支持**：QUIC 不再依赖 IP 地址和端口来标识连接，而是使用一个由服务器分配的 **Connection ID**。频繁切换网络情况下能保持通信，不会发生重连。
# HTTPS 和 HTTP 的区别，HTTPS 是如何保证安全的
| 对比维度 | HTTP | HTTPS |
| ---- | ---- | ---- |
| **安全性** | **明文传输**，数据容易被窃听、篡改 | **加密传输**，内容被加密，防窃听、防篡改 |
| **协议构成** | 直接基于 TCP 协议 | 在 HTTP 和 TCP 之间增加了 **SSL/TLS 层** |
| **证书** | 无需证书 | 需要向 CA 机构申请数字证书 |
| **性能** | 速度较快（无加密开销） | 稍慢（需要握手和加解密，但现代硬件已优化得很好） |
#### HTTPS
- **加密**：使用SSL/TLS协议对传输的数据进行加密，使用**非对称加密**在握手阶段安全地协商出**会话密钥**，然后使用**对称加密**来传输实际数据。保证了安全性。
- **身份验证**：通过**数字证书**体系，利用权威 **CA 机构** 的证书，浏览器可以验证服务器的身份，防止中间人攻击网站。
- **完整性校验**：HTTPS 利用 **Hash 算法**生成数据摘要，接收方通过比对摘要来验证数据的**完整性**，一旦发现篡改立即终止连接
# HTTP 状态码
- **1xx**（信息性状态码）：表示请求已被接收，继续处理。
	100 Continue
- **2xx**（成功状态码）：表示请求已成功处理。
	200 OK、201 Created
- **3xx**（重定向状态码）：表示请求需要进一步的操作（如跳转）。
	301 Moved Permanently(永久重定向)、302 Found(临时重定向)
- **4xx**（客户端错误状态码）：表示请求有错误，由客户端引起。
	400 Bad Request、404 Not Found、403 Forbidden
- **5xx**（服务器错误状态码）：表示服务器在处理请求时发生错误。
	500 Internal Server Error、502 Bad Gateway
# 常见 HTTP 请求方法
- **GET**：用于请求获取资源。
- **POST**：用于提交数据，通常用于创建或修改资源。
- **PUT**：用于更新资源。
- **DELETE**：用于删除资源。
- **HEAD**：类似于GET，但只请求头部。
- **OPTIONS**：用于获取支持的请求方法或查询服务器支持的选项。
- **PATCH**：用于部分更新资源。
# 强缓存/协商缓存
#### 强缓存
**逻辑**：只要在缓存有效期内，浏览器根本不会向服务器发送请求，直接从本地读取资源。
**机制**：服务器返回资源（状态码 200），并在响应头中带上缓存规则（如 `Cache-Control: max-age=3600`）。浏览器下载资源，并将其连同缓存规则一起存入本地缓存
*强缓存响应头*
* **Cache-Control**，一个多功能字段

| 指令 | 含义 |
| ---- | ---- |
| **`max-age=3600`** | **缓存有效期**。表示资源在 3600 秒（1小时）内有效。 |
| **`public`** | **公共缓存**。资源可以被浏览器、CDN、代理服务器等任何中间节点缓存。 |
| **`private`** | **私有缓存**。资源只能被浏览器缓存，代理服务器（如 CDN）不能缓存。 |
| **`no-cache`** | **跳过强缓存**。不使用本地缓存，每次都要去服务器验证（走协商缓存）。 |
| **`no-store`** | **禁止一切缓存**。完全不缓存，每次都重新请求。 |
* **Expires**，早期的缓存控制字段，指定一个**绝对过期时间**（例如 `Expires: Wed, 21 Oct 2025 07:28:00 GMT`），缺点：它依赖客户端的系统时间。如果用户修改了电脑时间，缓存判断就会出错。
* 优先级：Cache-Control 优先于 Expires
#### 协商缓存
**逻辑：** 当强缓存失效（过期）后，浏览器会向服务器发送请求，服务器根据资源是否修改来决定是返回新资源（200）还是告诉浏览器继续用旧缓存（304(协商缓存命中)）
核心字段：
*  **ETag**：是资源的唯一标识符，服务器通过返回ETag值判断资源是否发生变化。
- **If-None-Match**：客户端在请求时携带ETag，服务器判断是否存在差异，如果没有则返回304 Not Modified。
- **Last-Modified**：指定资源的最后修改时间。
- **If-Modified-Since**：客户端在请求时携带Last-Modified，服务器判断资源是否已被修改，如果未修改返回304。
- 优先级： 当 ETag 和 Last-Modified 同时存在时，服务器会优先使用 ETag 进行验证。
# 输入URL会发生什么
1. **DNS解析**：浏览器会先解析URL中的域名，获得服务器的IP地址。
2. **TCP连接**：建立传输层连接，如果是TCP，通过三次握手（如果是HTTPS，还会进行SSL/TLS握手）建立TCP连接。
3. **发送请求**：浏览器根据请求方法（GET、POST等）将数据发送到服务器。
4. **服务器响应**：服务器处理请求，返回状态码和响应数据。
5. **渲染页面**：浏览器根据响应的HTML、CSS、JS等文件进行页面渲染。
6. **关闭连接**：TCP连接通过四次挥手被关闭。
# 跨域同源策略
 浏览器为保证安全性，规定不同域之间的网页不能互相访问。只有在协议、域名、端口都相同的情况下，才在同一个域中，才能进行交互。否则就是跨域。目的是保护数据安全，防止跨站请求伪造和数据窃取。
 *跨域解决方法：*
  * **CORS:** 服务器通过设置 `Access-Control-Allow-Origin` 响应头，明确告诉浏览器允许哪些源访问。
  * **JSONP:** `<script>` 标签的 src 不受同源策略限制, 因此可以动态创建标签并在src中传递回调函数名来获取数据。
  * **代理：** 配置 Webpack/Vite 的代理，代理拦截浏览器的请求后，将跨域请求转发给后端
# 三次握手四次挥手
**三次握手**（建立连接）：
1. **SYN**：客户端发送SYN包请求建立连接。
2. **SYN-ACK**：服务器收到SYN包后，回复SYN-ACK包。
3. **ACK**：客户端收到SYN-ACK包后，回复ACK包，建立连接。

 **四次挥手**（断开连接）：
1. **FIN**：客户端发送FIN包，表示请求关闭连接。
2. **ACK**：服务器收到FIN包后，回复ACK包。
3. **FIN**：服务器准备好关闭连接后，发送FIN包。
4. **ACK**：客户端收到服务器的FIN包后，回复ACK包，连接关闭。
